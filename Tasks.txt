SwiftAccessibilityAgent — Research + Documentation Plan
======================================================
Date: 2026-03-07
Last Updated: 2026-03-07

Goal
----
Create a citable, source-driven Accessibility Best Practices knowledge base for SwiftUI and UIKit (including interop where it materially affects accessibility).
The deliverable should be directly usable by AI coding tools (Agent Skills style), and readable by humans.

Execution Defaults
------------------
- Owner default: `@agent` unless explicitly reassigned.
- Status values: `Not Started`, `In Progress`, `Blocked`, `Done`.
- Priority values: `P0`, `P1`, `P2`.
- Date format: `YYYY-MM-DD`.

Non-negotiables
---------------
- Prefer primary sources (Apple Developer Documentation, Apple HIG, WWDC sessions) when defining platform truth.
- Every guideline must cite at least one primary (Tier-1) source.
- If a guideline is derived from non-Apple sources, it must be anchored to Apple docs and/or framed as "practice" instead of "platform truth."
- Keep guidance testable: each guideline must include a verification method (Accessibility Inspector, VoiceOver walkthrough, UI tests, etc.).

Citation Quality Rules
----------------------
- Minimum per guideline: 2+ citations, including >=1 Tier-1 source.
- Include `Last Verified` date for each guideline and each source reference set.
- Prefer canonical URLs (Apple docs page root for API/topic, not mirror pages).
- Use direct quotes only when they add clarity; max 1-2 quotes per guideline and <=25 words per source.

Definition of Done (Global)
---------------------------
A task is `Done` only when all are true:
- Required artifact file exists at the specified path.
- Required citations are present and meet Citation Quality Rules.
- Verification steps are written and executable by a reviewer.
- Review pass completed (content quality + consistency + links check).
- `Last Updated` metadata is current.

Scope Targets
-------------
- SwiftUI track: semantics (labels/values/hints/traits/actions), grouping, focus/navigation order, custom controls, dynamic type, reduced motion, contrast, images/icons, form fields, lists, sheets, alerts.
- UIKit track: equivalent semantics and controls for UIKit-first apps (views, controls, containers, tables/collections, modal flows, media controls).
- Tooling: Xcode Accessibility Inspector; manual device testing with VoiceOver.
- Interop: `UIHostingController`, representables, UIKit containers affecting accessibility tree.

Risk Log
========
R1 - API Drift / Deprecation
- Impact: outdated modifiers or behavior guidance.
- Mitigation: quarterly Apple source review + changelog entries.

R2 - Source Conflict (Tier-1 vs community)
- Impact: inconsistent recommendations.
- Mitigation: Tier-1 always wins; mark community guidance as practice.

R3 - Untestable Guidelines
- Impact: guidance cannot be enforced in review.
- Mitigation: enforce Inspector + VoiceOver checks in every guideline.

R4 - Ambiguous Interop Behavior
- Impact: accessibility tree differs between SwiftUI-only and UIKit-hosted contexts.
- Mitigation: require interop notes and on-device checks for interop topics.

Review Cadence
==============
- Weekly status review: every Friday.
- Milestone review: at completion of each M1-M5 target.
- Freshness review: quarterly (Apple docs) + annually (WWDC/community).

Notes
=====
- Keep a hard boundary between platform truth (Apple docs) and best practice (community), but allow both with citations.
- Every guideline must tell the developer (or agent) how to verify correctness.
